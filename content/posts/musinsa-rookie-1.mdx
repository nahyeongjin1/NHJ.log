## **문제 1: 사이즈 추천**

---

### **문제 요약**

브랜드별 사이즈 정보(키, 가슴, 허리 범위)가 주어지고, 고객의 신체 치수를 기반으로 적합한 사이즈를 추천하는 문제였다.

### **출력 조건**

- 매칭되는 사이즈가 있으면 사이즈명 출력

- 브랜드가 없으면 `UNKNOWN`

- 모든 사이즈보다 크면 `UP`

- 모든 사이즈보다 작으면 `DOWN`

- 어느 사이즈에도 맞지 않으면 `MISMATCH`

### 접근 방식

1. 브랜드별로 사이즈 정보를 딕셔너리에 저장

1. 고객 요청마다 브랜드 존재 여부 → 범위 매칭 → 예외 케이스 순으로 분기 처리

1. 쉼표 기반 파싱으로 입력 처리

### 소요 시간

약 30분

## **문제 2: 스토어 픽업 예약**

---

### **문제 요약**

스토어별 운영 정보(영업시간, 픽업 한도)와 재고 정보가 주어지고, 픽업 예약 요청을 처리하는 문제였다.

### **실패 사유 (우선순위 순)**

1. `STORE` - 존재하지 않는 스토어

1. `TIME` - 운영 시간 외

1. `FULL` - 해당 시간대 픽업 한도 초과

1. `STOCK` - 재고 부족

### 접근 방식

1. 스토어 정보, 재고 정보, 시간대별 픽업 카운트를 각각 딕셔너리로 관리

1. 예약 요청마다 우선순위 순으로 검증

1. 성공 시 재고 차감 + 픽업 카운트 증가

### 기억나는대로 작성한 핵심 코드

```python
# 우선순위 순 검증
if store_id not in stores:
    print(f"{req_id},FAIL,STORE")
    continue

if not (store['open'] <= hour < store['close']):
    print(f"{req_id},FAIL,TIME")
    continue

if pickups[store_id].get(hour, 0) >= store['limit']:
    print(f"{req_id},FAIL,FULL")
    continue

if stocks[store_id].get(product_id, 0) < qty:
    print(f"{req_id},FAIL,STOCK")
    continue
```

## **실제 서비스 관점에서의 고민**

---

문제를 풀면서 들었던 생각은 두 문제 모두 무신사에서 실제로 다룰 법한 도메인이라는 점이다.

코딩테스트에서는 단순화된 버전이었지만, 실제 서비스에서는 훨씬 복잡한 문제들이 있을 것이다.

하나씩 생각해보면,

### 사이즈 추천 서비스

**코딩테스트에서는: **

- 브랜드 수 개, 사이즈 수 개

- 단순 범위 매칭

**실제 서비스에서는:**

- 수천 개 브랜드, 수백만 개 상품

- 같은 사이즈도 브랜드/상품별로 실측이 다름

- 사용자의 과거 구매 이력, 나와 비슷한 체형 데이터 활용

- 위 데이터를 기반으로 한 ML 기반 개인화 추천으로 발전 가능

**아키텍처 관점:**

- 상품별 실측 정보 캐싱 (상품 정보는 자주 바뀌지 않음)

- 추천 알고리즘 개선 시 A/B 테스트로 효과 검증 (추천 사이즈 클릭률, 사이즈 관련 반품률 등)

### 스토어 픽업 예약 시스템

**코딩테스트에서는:**

- 순차적 요청 처리

- 단일 서버 가정

**실제 서비스에서는:**

- 동시에 수천 명이 같은 상품 픽업 예약 시도

- 재고 정합성 문제 (Race Condition)

- 여러 채널(앱, 웹, 오프라인)에서 동시 재고 차감

**아키텍처적 관점:**

- 재고 차감은 strong consistency 필수 (overselling 방지)

- 분산 락 또는 DB 트랜잭션으로 동시성 제어

- K8s HPA를 통한 트래픽 기반 auto-scaling

- 예약 취소/변경 시 재고 복구 로직
